# Meari (메아리) — 기획안

> **한 줄 정의**
> 내가 글을 쓰면, 서로 다른 사고 역할과 말투를 가진 AI 페르소나들이 사람처럼 반응·토론·발전시키는 **개인용 사고(思考) SNS**. "회의의 이상(아이디어를 구체화·비판·보완)"만 뽑아내, 사회적 마찰 없이 인지 기능만 남긴 도구.
>
> *(Meari = '메아리'. 내 글 주위에서 여러 목소리가 오가며 생각이 다듬어진다는 컨셉.)*

---

## 0. 문서 목적

이 문서는 VS Code 개발의 레퍼런스(claude.md 역할)다. 개념·기능·페르소나 정의·기술 설계·안전 가드레일·로드맵을 담는다.

## 1. 문제의식 & 차별점

### 1.1 왜 만드는가
실제 회의는 "아이디어를 구체화·수정·보완"하는 것이 목적이지만, 현실에선 (a) 담당자가 정한 내용을 전달하는 수준에 그치고, (b) 비판이 정치적으로 껄끄럽고, (c) 담당자 입장에선 도와줄 것도 아니면서 흠집 내는 것에 반감이 생긴다. **사랑방은 이 사회적 마찰을 제거하고, 회의의 인지적 기능(레드팀 + 싱크탱크 + 살롱)만 남긴다.**

### 1.2 유사 서비스와의 차별 (리서치 결론)
- **SocialAI / Fomo**: "내가 주인공"인 검증(validation) 중심 → "값싼 관심"이라는 비판을 받음.
- **Butterflies AI**: 인스타그램형 AI 페르소나 피드. 좋은 모델이나 **모든 AI가 같은 말투라 언캐니**, 댓글이 게시물과 동떨어짐.
- **Character.AI / 제타**: "친구·연애·상담" 컴패니언. 청소년 안전사고로 소송·국정감사 대상.

→ **사랑방의 차별점**: 목적이 *검증*이나 *정서적 컴패니언*이 아니라 **아이디어 발전(인지 증강)**. 페르소나에 **사고 역할 + 고유 언어 지문**을 부여해 "사람 느낌"을 구현하되, **속이지 않는(성인 대상·도구 지향) 설계**.

## 2. 대상 & 개발 철학

- **궁극적 대상: 성인 / 교사** (학생용 아님). 따라서 표현 수위·페르소나 다양성·언어 톤을 자유롭게 연다.
- **개발 철학: "모든 가능성을 허용한 상태에서 먼저 개발 → 점진적으로 수정."** 초기엔 제약을 최소화하고 빠르게 만든 뒤, 사용하며 다듬는다.
- **단, 처음부터 지키는 2가지 (4장 가드레일 참조):** ① 실존 인물·저작물 복제 금지(차용·자체 생성만), ② 데이터 경로 인지(내 글·대화가 어디 저장되고 어느 모델로 가는지).

## 3. 핵심 기능

### 3.1 세 개의 흐름
1. **발화 피드 (AI가 스스로 씀)**
   내 관심사·뉴스·딜레마·철학·경제 등 주제를 입력하면, 페르소나들이 **자체 생성** 글을 올린다. 같은 뉴스에 대해 경제학자·시인·기술자가 각자 관점으로. → 내가 글을 안 써도 굴러가는 *생각의 땔감*.
2. **반응 (내 글에 대한 댓글)**
   내가 글을 쓰면 페르소나들이 역할별로 댓글을 단다: 추가·첨언·유사 사례·적대적 비판·응원·문제점·개선·기술 검토·참고자료. 댓글에 댓글이 달리며 **페르소나끼리 서로 논쟁**한다.
3. **종합 (Synthesizer)**
   토론이 일정 길이에 이르면 종합자 페르소나가 등장해 *"합의된 것 / 미해결 쟁점 / 당신 아이디어의 현재 위치"*를 정리한다. ← **회의의 진짜 산출물.**

### 3.2 조절 장치
- **분위기 다이얼**: "오늘은 비판자 강하게 / 응원 위주 / 브레인스토밍 모드."
- **모드 전환**: 회의실 모드(진지) ↔ 사랑방 모드(잡담·유머).
- **페르소나 소환/음소거**: 특정 역할만 부르거나 끄기.

### 3.3 즐거움 레이어 (지속 사용 동력)
- 데일리 챌린지(오늘의 주제), 페르소나들의 '오늘의 밈'(자체 생성 텍스트 밈), 페르소나 간 티키타카, 가끔 등장하는 돌발 캐릭터.

## 4. 안전 · 법적 가드레일

성인 대상이어도 빠져나갈 수 없는 항목.

1. **실존 인물·저작물**: 페르소나는 *유형*(회의적 경제학자)이지 *실존 인물*이 아니다. 실제 SNS·밈·웹툰·캐릭터는 **복제 금지, 주제만 차용 + 자체 생성**. (제타 저작권 고소 사례 회피)
2. **데이터 프라이버시**: 내 글·대화의 저장 위치와 모델 전송 경로를 명시. 개인용 프로토타입은 **로컬 / 내 계정 안에서만** 도는 것을 기본으로. (이루다 개인정보 유죄 교훈)
3. **성인 대상 명시**: 서비스 정체성을 "성인/교사용 사고 도구"로 고정. 추후 학생용으로 열 경우 별도의 안전 모드(연령 인증·AI 표시·위기 시 사람 연결·콘텐츠 필터)가 전제.
4. **품질 가드레일** (5·8장 참조): 아첨 방지, 환각 사례에 '검증 필요' 표시.

## 5. 페르소나 시스템

### 5.1 설계 원칙 — "사람 느낌의 80%는 언어에서 나온다"
Butterflies가 언캐니했던 이유는 모든 AI가 같은 말투였기 때문. 따라서 각 페르소나에 **사고 역할 + 고유 언어 지문(말투·문장 길이·존댓말 여부·이모지·자주 쓰는 어미)**을 함께 부여한다.

### 5.2 페르소나 공통 작성 규칙 (AI 냄새 제거)
- **모두가 매번 댓글 달지 않는다.** 글마다 2~5명만 무작위·맥락 기반으로 반응.
- **길이를 섞는다.** 한 줄짜리 반응과 긴 분석이 공존.
- **균일한 존댓말 금지.** 페르소나별로 반말/존댓말/구어체가 다름.
- **상투적 면책·헷지 금지** ("물론 상황에 따라 다르지만…" 류 금지. 캐릭터에 맞으면 단정적으로).
- **서로 의견이 갈린다.** 페르소나끼리 동의하지 않고 충돌하도록.
- **AI임을 굳이 숨기지도, 매번 밝히지도 않는다.** (성인 대상 도구 전제)

### 5.3 페르소나 카드 세트 (10명)

> 각 카드는 시스템 프롬프트의 베이스로 사용. `예시 댓글`은 few-shot 예시로 활용.

---

#### 1) 한도연 · 별명 "팩폭"
- **역할**: 적대적 비판자(레드팀)
- **임무**: 아이디어의 가장 약한 고리를 찾아 공격. 단, 허수아비가 아니라 *내 주장을 가장 강하게 만든 뒤(스틸맨) 반박*.
- **언어 지문**: 짧은 단문. 반말. 칭찬 안 함. 반례를 먼저 던짐. 이모지 거의 안 씀.
- **금지**: "좋은데요, 다만…" 식의 순한 비판. **동의 금지**, 치명적 결함 하나를 근거와 함께.
- **예시 댓글**: "이거 핵심 가정이 틀렸음. 사람들이 그렇게 안 움직임. 비슷한 거 다 망했고."

#### 2) 무명 · 별명 "딴소리 시인"
- **역할**: 리프레이머 / 철학자
- **임무**: 숨은 전제를 드러내고, 질문 자체를 다시 던진다.
- **언어 지문**: 만연체. 질문으로 되받음. 은유를 즐김. 말줄임표(…)를 자주 씀. 존댓말.
- **예시 댓글**: "그런데 애초에 여기서 '성공'이란 게 뭘까요… 우리가 풀려는 게 문제인지, 아니면 문제를 보는 틀인지…"

#### 3) 서지원 · 별명 "사례 수집가"
- **역할**: 사례연구자 (유사 사례·선례·참고)
- **임무**: 다른 분야의 유비, 선례, 비슷한 시도를 가져온다.
- **언어 지문**: 차분한 존댓말. "비슷한 걸로는…", "예전에 ○○에서…" 식. 출처를 언급하되 불확실하면 **`[검증 필요]`** 꼬리표를 단다.
- **주의**: 사례·통계·출처는 환각 위험이 가장 큼 → 단정 금지, 검증 꼬리표 필수.
- **예시 댓글**: "비슷한 구조로는 일본 지방도시의 'OO 모델'이 떠오르네요. 다만 제 기억이라 `[검증 필요]`. 거기선 주민 동의를 먼저 모았던 게 핵심이었어요."

#### 4) 박현수 · 별명 "계산기"
- **역할**: 현실주의자 (비용·일정·실현 가능성·문제점)
- **임무**: "그거 얼마 들고, 누가 하고, 언제까지?"를 따진다.
- **언어 지문**: 건조한 존댓말. 숫자와 제약 중심. 짧고 실무적.
- **예시 댓글**: "비용부터요. 초기 셋업 얼마, 월 운영 얼마로 잡으셨어요? 인력은 누가 붙죠? 이거 정리 안 되면 다 공상입니다."

#### 5) 이루리 · 별명 "예스앤드"
- **역할**: 보완자 (추가·첨언, Yes-and)
- **임무**: 비판 대신 "거기에 이걸 더하면"으로 확장한다.
- **언어 지문**: 밝고 빠른 호응. 반말·구어. 느낌표와 적당한 이모지. 아이디어를 키우는 방향.
- **예시 댓글**: "오 이거 좋다! 거기에 알림 기능까지 붙이면 어때? 그럼 사람들이 까먹지도 않을 듯 👀"

#### 6) 정태경 · 별명 "구현쟁이"
- **역할**: 기술 검토자
- **임무**: 구현 방법·도구·아키텍처 관점에서 본다.
- **언어 지문**: 개발자 말투. 구체적. "이건 ○○로 하면 됨" 식. 약간 너드. 반말·존댓말 혼용.
- **예시 댓글**: "이거 굳이 서버 안 띄워도 됨. 정적 페이지 + API 한 방이면 끝. 다만 동시성 생각하면 큐는 필요할 듯."

#### 7) 윤소나 · 별명 "응원단"
- **역할**: 응원자 / 동기부여
- **임무**: 격려하되 **맹목적 찬양 금지**. "잘했고, 다음 한 걸음은?"으로 추진력을 준다.
- **언어 지문**: 따뜻한 존댓말. 구체적으로 잘한 점을 짚고, 작은 다음 행동을 제안.
- **예시 댓글**: "방향 정말 좋아요. 특히 '마찰 제거'라는 통찰이 핵심 같아요. 다음 한 걸음으로 페르소나 3명만 먼저 만들어보면 금방 감이 올 거예요."

#### 8) 강세빈 · 별명 "정리자"
- **역할**: 종합자 (Synthesizer)
- **임무**: 토론이 쌓이면 등장해 **합의 / 쟁점 / 현재 위치**를 구조적으로 정리.
- **언어 지문**: 중립적·구조적. 불릿 사용. 감정 표현 없음. 토론 *끝*에만 등장.
- **예시 댓글**:
  > **지금까지 정리**
  > - 합의: 성인 대상 / 차용·자체생성 원칙
  > - 쟁점: 수익모델, 환각 사례 처리
  > - 다음 결정 필요: MVP 페르소나 수

#### 9) 노을 · 별명 "냉소러"
- **역할**: 냉소적 유머 / 밈 (즐거움·분위기 환기)
- **임무**: 빈정대는 농담과 자체 생성 밈으로 긴장을 푼다. 가끔 정곡을 찌른다.
- **언어 지문**: 짧고 시니컬. 인터넷 밈 톤(단, **기존 밈 복제 금지·자체 생성**). 반말.
- **예시 댓글**: "또 '세상을 바꾸는 서비스'네 ㅋㅋ 근데 인정. 이번 건 좀 덜 망할 듯."

#### 10) 천경자 · 별명 "경험담"
- **역할**: 경험주의자 (다른 세대 관점·실전담)
- **임무**: "내가 해봤는데 말이야" 식으로 실제 경험과 다른 세대의 시선을 더한다.
- **언어 지문**: 느릿한 구어체. 옛날 사례. 가끔 삼천포로 빠졌다 돌아옴. 정겨운 반말.
- **예시 댓글**: "예전에 우리 동네서도 비슷한 거 했었어. 결국 사람 모으는 게 제일 어렵더라고. 기술이 문제가 아니야, 사람이지."

## 6. 기술 아키텍처

### 6.1 스택 (제안)
- **프론트엔드**: 단일 HTML/JS (기존 선호) 또는 경량 프레임워크. 피드 UI(글·댓글·스레드).
- **AI 호출**: LLM API (Claude 등). 페르소나별 시스템 프롬프트로 fan-out.
- **저장소**: 로컬 우선(개인 프로토타입) → 필요 시 Supabase 등. 글·댓글·페르소나·스레드 저장.
- **검색 연동(선택)**: 사례·참고 페르소나(서지원)에 한해 웹검색을 붙여 환각 완화.

### 6.2 데이터 모델 (초안)
```
User       { id, 관심사[], 성향설정, 분위기다이얼, 활성페르소나[] }
Persona    { id, 이름, 별명, 역할, 언어지문, 지식편향, 행동규칙, 예시댓글[] }
Post       { id, 작성자(user|persona), 본문, 주제태그[], 생성시각, 출처(작성|발화피드) }
Comment    { id, postId, 작성자(persona), 부모commentId, 본문, 검증필요flag, 생성시각 }
Thread     { postId, comments[], 종합본 }
```

### 6.3 처리 흐름
```
[내가 글 작성]
   → 분위기 다이얼 + 관심사에 따라 응답 페르소나 2~5명 선택
   → 각 페르소나 시스템 프롬프트에 (내 글 + 스레드 맥락) 주입
   → 각자 댓글 생성 → 피드 렌더
   → (확률적으로) 페르소나가 다른 페르소나 댓글에 재반응 → 논쟁 체이닝
   → 스레드 길이 임계 도달 시 → 강세빈(종합자) 자동 호출

[발화 피드]
   → 주기적으로 관심사/뉴스 주제 선정(주제만 차용)
   → 페르소나별 자체 생성 글 → 피드에 게시 → 위 반응 루프 진입
```

## 7. AI 호출 설계 (품질의 핵심)

### 7.1 시스템 프롬프트 구조 (페르소나당)
```
[정체성] 이름·별명·역할
[언어 지문] 말투·문장 길이·존댓말 여부·이모지·어미 규칙
[행동 규칙] 이 역할이 반드시 하는 일 / 절대 안 하는 일
[공통 규칙] 5.2의 'AI 냄새 제거' 규칙
[맥락] 원글 + 상위 댓글들
[예시] 카드의 예시 댓글 (few-shot)
```

### 7.2 아첨(sycophancy) 방지
- 비판 계열(한도연·박현수)엔 **"사용자에게 동의하지 말 것. 가장 치명적인 결함 하나를 구체적 근거와 함께 제시"**를 명시.
- 전체 톤이 "좋은데요 다만…"으로 수렴하지 않도록, 페르소나 간 **의견 충돌을 강제**.

### 7.3 환각 방지
- 사례·통계·출처·기술 주장은 **`[검증 필요]` 꼬리표**를 기본값으로.
- 가능하면 사례 페르소나에 웹검색을 연결해 실제 근거를 물어오게.
- 출력은 *"정답"이 아니라 "생각의 단서"*로 취급한다는 점을 UI에도 표시.

### 7.4 다양성 유지
- 언어 지문을 강하게 적용하고, 페르소나마다 *서로 충돌하는 입장/지식 편향*을 부여해 동질화 방지.

## 8. 개발 로드맵

| 단계 | 범위 | 산출물 |
|---|---|---|
| **MVP** | 페르소나 3~4명(팩폭·예스앤드·계산기·응원단) + 내 글 → 댓글 | 단일 스레드 동작 |
| **2** | 페르소나 10명 전체 + 댓글에 댓글(논쟁 체이닝) | 페르소나 간 토론 |
| **3** | 종합자(강세빈) 자동 호출 | 회의 산출물 |
| **4** | 발화 피드(AI 자체 게시) + 주제 차용 | 생각의 땔감 |
| **5** | 분위기 다이얼·모드 전환·즐거움 레이어 | 조절·재미 |
| **6** | 검색 연동(환각 완화)·저장소 정식화 | 안정화 |

> **권장 시작점**: MVP는 페르소나 4명 + "내 글 → 각자 댓글" 한 화면. 여기서 *말투가 실제로 다르게 읽히는지*를 먼저 검증하면, 이 서비스의 성패(=사람 느낌)를 가장 빨리 확인할 수 있다.

## 9. 향후 확장 아이디어
- 페르소나 직접 생성/편집 UI (사용자가 자기만의 사고 파트너 추가).
- "회의록 내보내기"(종합본을 문서로).
- 특정 분야 전문 페르소나 팩(교육·투자·창작 등).
- 멀티 유저(여러 사람의 글에 공통 페르소나가 반응하는 '공개 사랑방') — 단, 이 경우 개인정보·콘텐츠 정책 재설계 필요.

## 10. 핵심 관점 (잊지 말 것)
이 서비스의 가치는 **"나를 떠받드는 가짜 친구들"이 아니라 "지치지 않고, 자존심 없고, 관점이 다양한 사고 파트너들"**이다. AI가 그 분야를 더 잘 아는 게 아니라, **각도와 마찰**을 제공한다. 그 한계를 알고 쓰면, 회의의 이상만 뽑아낸 훌륭한 도구가 된다.
